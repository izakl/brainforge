# Learning: Onboarding learnings from Acme (first managed brain)

- From brain: izakl/acme-autonomy-system
- Framework version seen: 0.1.0
- Proposed core module: onboarding-engine (with task-queue and inspect-first notes)
- Kind: changed

## What we observed

Onboarding Acme end-to-end (adopt-existing, inspect-first) surfaced several
reusable patterns:

- **Inspect-first worked.** The gap report flagged 4 HIGH external
  source-of-truth risks (the master context lived Drive-only) and a
  PostgreSQL 16-vs-17 drift between app docs and `docker-compose` — both
  actionable before any code changes.
- **Non-text artifacts are unsafe through the text reader.** A JSX/React file
  came through the Drive *text* reader with Markdown characters escaped
  (`#`, `*`, `>`, `[` ...), corrupting the code. The *base64* download was
  faithful, but reproducing ~30 KB of base64 by hand truncated a long
  repetitive block. The reliable pattern: download base64, decode, **assert
  the exact byte size**, and retry (in chunks) until it matches — then confirm
  distinctive content survived.
- **Repo rename needs a cutover discipline.** Renaming `legacy_za` to
  `acme` meant preserving live GitHub references (`gh` commands, repo URLs)
  until the operator performed the rename, then reconciling them afterward.
- **The task-queue drift rule cascades `done`.** A task completed out-of-band
  cannot be marked `done` without its dependent immediately becoming a
  dependency-ready `pending` task (auto-prepared on the next merge). We kept
  `onboard-acme-first-brain` at `pending` so finishing Acme early did not
  auto-queue the next project's work.

## What we propose for the framework

Project-neutral improvements:

1. **Onboarding engine:** codify that non-text artifacts (code, binaries,
   images) migrate via **base64 + byte-exact verification**, never the
   text/Markdown reader; add a verify-and-retry loop to the migration step.
2. **Inspect-first:** make "version/config drift between app docs and runtime
   config" a standard inspector check (it caught the Postgres drift here).
3. **Task-queue model:** provide a way to record **out-of-band completion**
   without forcing the dependent task to be auto-prepared — e.g. a
   `done`-without-auto-flow marker or an explicit operator `hold` — so finishing
   work ahead of the queue does not prematurely queue the next project.
